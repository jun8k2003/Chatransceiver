## [DEC-003] 送信排他制御：非同期キュー（順次再生）方式の採用
* **日付**: 2026-06-01
* **決定事項**: 送信権の排他制御は行わず、同時に送信された音声はチャット画面に順番に並び、受信側では届いた順に1つずつ自動再生（順次再生）される「非同期キュー」方式を採用する。
* **背景・理由**: 「誰かが話しているから送信できない」というストレスを無くし、チャットUIの非同期の利点（ログが残り、自分のタイミングで送れる）とトランシーバーの利点（自動的に声が流れる）を最も良く融合できるため。

## [DEC-004] ログイン/接続アクションによる自動再生ロックの解除
* **日付**: 2026-06-01
* **決定事項**: アプリ起動時に「ログイン」または「チャンネル接続」のボタンをクリックするフローを必須とし、このクリックイベントをもってブラウザの自動再生ポリシー（音声ブロック）をクリアする。
* **背景・理由**: チャット画面に入った後はユーザーに余計なタップを強いることなく、最初期の着信から完全にハンズフリーで自動再生されるトランシーバー体験を実現するため。

## [DEC-005] 認証・データベース・ストレージ基盤としての Supabase の採用
* **日付**: 2026-06-01 (更新)
* **決定事項**: ユーザー認証、リアルタイムデータベース、および音声ファイルストレージに **Supabase** を採用する。（当初検討していた Firebase から移行）
* **背景・理由**:
  * Firebase Auth と同様に、Googleログインなどの認証機能を無料で容易に統合できること。
  * チャットアプリの性質上、将来的に「ユーザーのグループ所属」「既読管理」「メッセージ履歴」などの複雑な関係性（リレーショナルデータ）を扱うことになり、NoSQL（Firestore）よりも SQL（SupabaseのPostgreSQL）の方が設計上・拡張上のメリットが大きいこと。
  * 認証情報とデータベースの Row Level Security (RLS) が密接に統合されており、セキュリティ設計がシンプルになること。

## [DEC-006] 音声フィードバック用のマイク入力レベルメーターの実装
* **日付**: 2026-06-01
* **決定事項**: 録音ダイアログに、入力音声を視覚的にフィードバックする「マイク入力レベルメーター（音量インジケータ）」を実装する。

## [DEC-007] 無料構築を優先したWeb Speech APIの採用とマイク選択機能のオミット
* **日付**: 2026-06-01
* **決定事項**: ディクテーションは完全無料で動作するブラウザ標準の **Web Speech API (SpeechRecognition)** を使用する。これに伴うデバイス競合を避けるため、アプリ内での明示的な「マイクデバイス選択」機能は見送り、OS/ブラウザのデフォルトマイクを使用する仕様とする。
* **背景・理由**: 初期フェーズの実証実験においてコストを完全に無料に抑えつつ、競合エラーの発生を防ぎ安定して音声認識を動作させるため。

## [DEC-008] マイク権限エラー時の対処UI（設定URLコピー案内）の採用

* **背景・理由**: ユーザーが自分で素早くパーミッション（権限）エラーを解決できるようにするため。

## [DEC-009] グローバルな着信音声自動再生と環境設定による制御の採用
* **日付**: 2026-06-02
* **決定事項**: ユーザーが所属するすべての個別チャット・グループチャットから届く新着音声を、現在開いているチャットウィンドウに関わらず自動再生（グローバルキューによる順次再生）する。ただし、ユーザーごとの環境設定で「着信メッセージの即時自動再生」がON（デフォルトはON）の場合のみ動作する。
* **背景・理由**: 「どこから話しかけられても即座に声が聞こえる」という本物のトランシーバー体験を実現するため。自動再生による意図しない音漏れや騒音を防止するため、環境設定でこれをOFFにできる逃げ道（オプトアウト）を用意する。

## [DEC-010] 耳での聴取と画面での既読の分離管理（画面表示による既読化）
* **日付**: 2026-06-02
* **決定事項**: 音声が自動再生（聴取）された場合でも、ユーザーがそのチャット画面を実際に開いて表示するまでは「未読」状態を維持する。チャット履歴を開いた時点で初めて「既読」状態へと更新する。
* **背景・理由**: 音声の自動再生だけでは聞き流してしまう（または内容を十分に咀嚼できない）可能性があるため、UI上に「未読バッジ」を残すことで、ユーザーが後から確実に履歴を確認し、メッセージバブルの▶ボタンで再再生できるようにするため。

## [DEC-011] メンバー構成をキーとするグループ管理（メンバー追加＝別グループ化の許容）
* **日付**: 2026-06-02
* **決定事項**: グループチャットID（ChatHistoryID）は、参加している「メンバーの組み合わせ」と1対1で対応する。そのため、既存のグループチャットに新たなメンバーを追加（または削除）した場合、過去の履歴は引き継がれず、自動的に「新しい別のグループチャット」として処理される仕様とする。
* **背景・理由**: グループ管理のデータベース設計を「ダイレクトメッセージ（複数人宛先）」の拡張としてシンプルに保ち、明示的なグループ作成・編集の手間を省くため。この挙動による「メンバー増減時に別グループになる」制約をトレードオフとして許容する。

## [DEC-012] 個別一覧とグループ一覧 of チェックボックス状態の相互同期（排他選択の廃止）
* **日付**: 2026-06-02
* **決定事項**: 個別チャット一覧（左ペイン）とグループチャット一覧（中央ペイン）のチェック状態は排他とせず、構成メンバー情報をキーとして**双方向に自動同期（連動）**させる。
  * **グループ選択時**: グループを選択すると、個別一覧の該当構成メンバーのチェックが自動的にONになる。
  * **複数個別メンバー選択時**: 個別一覧でチェックされたメンバーの組み合わせに一致するグループが既に存在する場合、グループ一覧の該当グループのチェックが自動的にONになる。一致するグループがない場合はグループのチェックはすべてOFFになる。
* **背景・理由**: 「排他選択」にするよりも、「選択されたメンバーの集合」という単一の状態で管理する方がUIとして直感的に理解しやすく、グループの実体（誰が含まれているか）が常に視覚的にクリアになるため。

## [DEC-013] ニックネーム制と連番ユーザー番号による識別＆プライバシー管理の採用
* **日付**: 2026-06-02
* **決定事項**: 個別一覧やチャット上でメールアドレスは非表示とし、初回ログイン時に設定させる「ニックネーム」で表示する。また、同名回避と識別のため、システム（またはコミュニティ内）への参加順に `1` から始まる連番の「ユーザー番号」を自動付与し、UI上は `#1 ニックネーム` のように表示する。
* **背景・理由**: 個人情報（メールアドレス）の露出を防ぎつつ、同じニックネームが登録された場合でもユーザー同士が確実に相手を識別できるようにするため。

## [DEC-014] コミュニティ（招待コード/ワークスペース）切り替え方式の採用
* **日付**: 2026-06-02
* **決定事項**: ユーザー同士の繋がりとプライバシー管理の境界として「コミュニティ」の概念を導入し、画面上で現在アクティブなコミュニティを切り替える方式を採用する。
  * **制限範囲**: 個別一覧（左ペイン）およびグループ一覧（中央ペイン）には、選択中のアクティブなコミュニティに所属するメンバー/グループのみを表示する。
  * **参加方法**: 招待コードを用いてコミュニティに参加する。
  * **ユーザー番号の割り当て**: ユーザー番号（連番）は「コミュニティへの参加順」で1から付与する。
* **背景・理由**: 多数のユーザーが参加しても「知らない第三者」が一覧に表示されるのを防ぎ、家族・職場・趣味などの異なるコミュティ（コンテキスト）の境界をクリアに分離するため。

## [DEC-015] URLベースの招待リンク機能と英数字コミュニティID（スラッグ）の採用
* **日付**: 2026-06-02
* **決定事項**: コミュニティへの招待プロセスを容易にするため、URLリンクによるワンクリック招待機能を実装する。
  * **英数字コミュニティID（スラッグ）**: コミュニティ作成時に、URLに含められるユニークな英数字の識別子（スラッグ）をユーザーに設定させる（例: `team-a`）。
  * **招待リンク仕様**: 招待URLは `https://[domain]/c/[community_id]/join?code=[invite_code]` などの形式とし、招待コードをクエリパラメータに含める。
  * **参加フロー**: リンクにアクセスしたユーザーが未ログインならGoogleログイン後、ログイン済なら即座に「参加確認ダイアログ」を表示し、接続ボタン押下でコミュニティへ自動追加する。
* **背景・理由**: 手動で長い招待コードを入力する手間を省き、URLをメッセージやメールで共有するだけで即座にチームメンバーをコミュニティへ招待・合流できるようにするため。

## [DEC-016] コミュニティからの退出機能と参加時の同意確認ダイアログの採用
* **日付**: 2026-06-02
* **決定事項**: ユーザーが自律的に所属を管理できるようにするため、コミュニティからの「退出（脱退）機能」を実装する。また、招待リンクアクセス時に自動参加とせず、必ず「参加の意思を確認・同意するダイアログ」を挟むフローとする。
* **背景・理由**: 勝手に知らないグループのメンバーリストに追加されるのを防ぐセキュリティと、不要になったコミュニティとの接続をユーザー自身の意思でいつでも切断（退出）できるようにするため。

## [DEC-017] 編集可能コンボボックスによるコミュニティ接続と履歴のLocalStorage保存
* **日付**: 2026-06-02
* **決定事項**: コミュニティの選択・入室インターフェースとして、画面最上部（トップレベル）に **「編集可能なコンボボックス（ドロップダウン付きテキスト入力）」** を採用する。
  * **初期アクセス**: URLパラメータ（例: `?c=community-slug`）がある場合は直接そのコミュニティの接続を試み、パラメータがない場合は空白（未接続）の状態でアプリを開く。
  * **入室履歴の保存**: 入室に成功したコミュニティID（英数字スラッグ）は、ローカル環境（LocalStorage）に自動保存する。
  * **選択挙動**: ユーザーはコンボボックスに手動で英数字IDを直接入力することも、ドロップダウンから過去に入室成功した履歴をクリックして選択することもできる。
* **背景・理由**: コミュニティへの接続（入室）操作をキーボード入力と選択の両面でシームレスに行えるようにし、過去の入室履歴を手間なく呼び出せるようにするため。

## [DEC-018] テキストメッセージ送信とWeb Speech API (SpeechSynthesis) による音声合成 (TTS) 再生の採用
* **日付**: 2026-06-02
* **決定事項**: 音声発話（録音）が困難な環境を考慮し、テキストキーボード入力によるメッセージ送信機能を追加する。
  * **データの区別**: テキスト由来のメッセージは音声ファイルURLを持たない形でDBに保存する。
  * **再生仕様**: テキスト由来 of メッセージを着信自動再生、またはチャットバブルの「▶」ボタンで手動再生する際は、ブラウザ標準の **Web Speech API (SpeechSynthesis)** を使用してテキストを音声合成で読み上げる。
* **背景・理由**: 静かな環境にいて声を出せないユーザーであってもチャット感覚で発言でき、受信側にはトランシーバーの使い勝手（音声自動再生）を損なわずに声を届けるバリアフリーな体験を提供するため。

## [DEC-019] 開発言語としての TypeScript および ビルド環境としての Vite の選定
* **日付**: 2026-06-02
* **決定事項**: UIフレームワークは使用せず、開発言語として **TypeScript (TS)**、CSSはプレーンな **Vanilla CSS** を使用する。プロジェクトのトランスパイル・バンドル・ローカルデバッグ実行環境には **Vite** を採用する。
* **背景・理由**:
  * TypeScriptを採用することで、複雑なデータモデル（ChatHistoryIDなど）や再生キューの設計における型安全性を確保し、バグを未然に防ぐため。
  * **Vite** は React/Vue などのフレームワーク専用と思われがちだが、実際は Vanilla HTML/TS に対する最速のビルドツールであるため。設定ファイルがほぼ不要で、TSの即時トランスパイル、HMR（自動ホットリロード）、ソースマップによるブラウザ上での容易なTSデバッグ、および本番用バンドルが最初から揃っているため、最も開発効率が高くなると判断。

## [DEC-020] Google OAuth専用化と堅牢なセキュリティ・ルールの再設計 (RLS簡素化)
* **日付**: 2026-06-02
* **決定事項**: 
  1. ユーザー認証の仕組みを「メールアドレス＆パスワード入力」から「Google OAuth連携のみ」へと完全移行し、UIも「Googleでログイン」ボタンのみへ簡素化する。
  2. データベース（Supabase）の Row Level Security (RLS) を見直し、自己参照による無限ループ（再帰クエリ）のバグを排除するため、「認証済みユーザー（auth.role() = 'authenticated'）であればコミュニティやチャットルームの情報を読み書き可能」というシンプルで堅牢なポリシーへ再設計する。
* **背景・理由**:
  * メールアドレスやパスワードを自前で管理・入力させるフローは、UX的にも実装コスト的にも負担が大きく、Google連携であればワンクリックで安全にニックネームまで自動取得できるため。
  * 当初の厳密なRLSポリシー（「自分が所属するコミュニティのみ参照可能」など）は、同じテーブルを条件句で再帰的に参照することになり、PostgreSQLで無限ループによるサイレントエラーを引き起こし、参加しているはずのコミュニティデータが取得できない深刻なバグの温床となっていたため。これを解消し、システム全体を安定稼働させるため。

## [DEC-021] 個別チャット一覧での自分自身の非表示化
* **日付**: 2026-06-02
* **決定事項**: 左ペイン（個別チャット一覧）において、ログインしている自分自身をリストから除外して表示しない仕様とする。
* **背景・理由**: 一般的なチャットアプリにおいて「自分宛てのメッセージ」をメモ代わりに使うケースはあるものの、トランシーバー用途としての即時性や画面の視認性を重視した場合、自分自身が一覧にいることはノイズになるため。グループ作成時の宛先としても自分を含める必要がない（暗黙的に自分が含まれる）ため、UIをシンプルに保つ。

## [DEC-022] iOS等のプッシュ通知制限の代替としてDiscord Webhook連携の採用
* **日付**: 2026-06-06
* **決定事項**: iOS等のPWA非対応環境でWebプッシュ通知を受信できない問題の代替案として、ユーザー環境設定で個別にDiscord Webhook URLを登録し、Edge FunctionからFCMと並行してDiscordへも新着通知を送信する機能を実装する。
* **背景・理由**: Appleの仕様上、iOSブラウザでWebプッシュ通知を安定して受け取るにはPWA化（ホーム画面追加）などの高いハードルがあり、ITリテラシーに関わらず利用できるという要件に反するため。ユーザーが個別に通知のON/OFFを制御しやすく、かつ多くのユーザーが馴染みのあるDiscordへのWebhook送信をオプションとして提供する。
