# アプリケーション仕様書 (docs/specifications.md)

このファイルは、合意・確定した具体的な仕様を管理します。

## 1. UI・画面構成 (DEC-024 リデザイン後)
DEC-024 のリデザインにより、従来の3ペイン（個別／グループの上下分割）は**1つの統合リスト＋フィルタチップ**に集約されました。画面は最上部の操作ヘッダーと、その下のサイドバー（統合リスト）＋チャットウィンドウで構成します。

* **画面最上部 (ヘッダー)**:
  * **コミュニティ名ピル**: 接続中のコミュニティ名を表示。クリックで開くドロップダウンに「接続履歴・コミュニティID入力・招待リンクコピー・切断・退会」を集約 (DEC-024)。
  * **常時表示トグル (☀️)**: Screen Wake Lock 対応環境のみ表示 (DEC-026)。
  * **メディアボタンPTTトグル (🎧)**: 対応環境のみ表示 (DEC-027)。
  * **設定アイコン (⚙️)**: 環境設定モーダルを開く。
  * **ログイン状態表示**: 現在ログイン中のユーザー名（ニックネーム）を表示します。
* **サイドバー (統合リスト)**:
  * 個別チャット（メンバー）とグループチャットを**1つのリストに統合**して表示します。※自分自身は表示されません (DEC-021)。
  * **フィルタチップ**（すべて／個別／グループ）で表示を絞り込みます。
  * 行のタップ／クリックで即座に対応するチャットを開きます（上下スプリッターは廃止）。
  * **新規グループ作成**は「＋新規グループ」ボタンから入る専用の複数選択モードで行います。
  * メンバー表示形式は `# [コミュニティ内ユーザー番号] [ニックネーム]`（例: `#12 たろう`）。
* **チャットウィンドウ**:
  * 現在選択されている「チャット履歴ID」に紐づくチャット履歴（メッセージタイムライン）を表示します。
  * PTTボタンは、スマホでは下段中央の大型円形ボタン、PCではフッター左端の控えめなボタンとして配置されます (DEC-024)。

> **補足**: 以降のセクションで「左ペイン（個別）」「中央ペイン（グループ）」と記載している箇所は、リデザイン後は統合リスト内の**個別チャット行／グループチャット行という論理的な区別**を指します。構成メンバー情報をキーとした個別⇔グループの双方向同期 (DEC-012) のロジックはリデザイン後も維持されています。

## 2. データ構造と選択の挙動

### 2.1. チャット履歴ID (ChatHistoryID) による一元管理
* すべてのメッセージ履歴は、個別チャット・グループチャットの区別なく、ユニークな `ChatHistoryID` に紐づいてデータベース（PostgreSQL）で管理されます。

### 2.2. 個別チャット一覧（左ペイン）での選択挙動
* **1人のみ選択した場合**:
  * 「自分 ＋ 相手」の個別チャットに対応する既存 of `ChatHistoryID` が選択状態（ActiveChatHistoryID）になります。
  * この `ChatHistoryID` に基づき、チャット履歴ウィンドウ（右ペイン）が表示され、グループチャット一覧（中央ペイン）のチェックはすべてOFFになります。
* **自分以外の複数人（2人以上）を選択した場合**:
  * **既存グループが存在する場合**: 
    * 選択したメンバー構成に完全に一致する既存の `ChatHistoryID` が選択状態（ActiveChatHistoryID）になります。
    * このIDに基づき、中央ペインの該当グループのチェックボックスが自動的にONになり、チャット履歴が表示されます。
  * **既存グループが存在しない場合**:
    * 一時的に「新規作成状態（ActiveChatHistoryID ＝ null ＆ 宛先メンバー集合を保持）」となります。
    * 中央ペインのチェックはすべてOFFになり、チャットウィンドウには「**新規グループチャットを作成中...**」「メンバー: [ユーザー名リスト]」と視覚的に明記されます。
    * この状態で最初のメッセージを送信した時点で、新規に `ChatHistoryID` がDB上で生成され、それがアクティブなIDとして確定します。以降は、そのIDを基に各ペインが同期します。
* **メンバー変更時の挙動（制約）** (DEC-011):
  * グループの参加メンバーの組み合わせそのものが `ChatHistoryID` と1対1で対応するため、既存のグループにユーザーを追加（または削除）して発信すると、それは過去のIDとは異なる「新しい `ChatHistoryID`」の別グループとして扱われます。

### 2.3. グループチャット一覧（中央ペイン）での選択挙動
* グループをチェックすると、該当する既存の `ChatHistoryID` に紐づくチャット履歴が表示されます。
* 同時に、**そのグループの構成メンバー全員分について、左ペイン（個別チャット一覧）のチェックボックスが自動的にONになります** (DEC-012)。

### 2.4. グループ名（表示上のあだ名） (DEC-023)
* グループには任意の表示名（`chat_rooms.name`）を付けられます。**名前が未設定（NULL）の場合は、従来どおりメンバー名のカンマ羅列を表示**します。
* グループの同一性はあくまでメンバー構成で決まり（DEC-011）、名前は表示上のあだ名にすぎません。
* **メンバーなら誰でも編集可能**です（グループ削除と同じ権限思想）。
* 編集UI: チャットヘッダーのグループ名横の ✏️ ボタンをクリックするとインライン入力欄に変わり、Enter またはフォーカスアウトで確定、Esc でキャンセルします。

## 3. 画面遷移 & ユーザー体験 (UX) シナリオ

### 3.1. 使用開始フロー (ログイン・初期設定)
1. **初期ロード時のパラメータ判定**:
   * アプリ起動時、URLパラメータ（例: `https://[domain]/?c=community-slug`）をチェックします。
   * パラメータがある場合は直接そのコミュニティの入室（接続）処理を開始します。
   * パラメータがない場合は、コミュニティ接続コンボボックスが**「空白（未接続）」**の状態でアプリを開きます (DEC-017)。
2. **ログイン要求**: アプリ起動時、Supabase Auth によるログインを求めます。**Googleログイン**に加え、**Magic Link（メールリンク）認証**を選択できます (DEC-028)。同一の検証済みメールアドレスを使った場合、Google と Magic Link は同じ `auth.users.id` に集約されます。
3. **初回登録時（ニックネーム入力）**:
   * 初回アクセス時に、ユーザーに「ニックネーム（後から変更可能）」を入力させます。
   * ※メールアドレス等の個人情報は画面上に一切表示しません (DEC-013)。
4. **初期設定＆接続**: ログインおよびユーザー登録成功後、セッション用設定を行います。
   * 設定項目: 音声自動再生の有効/無効、マイクテスト（発声レベルメーター動作確認）、ニックネーム変更
   * コミュニティ接続: コミュニティIDを入力するか、LocalStorageから取得した履歴ドロップダウンから選択して接続ボタンをクリックします。
   * アクション: **「接続する（決定）」ボタン** をクリック。
   * ※このクリックによってブラウザの自動再生ブロックを解除します (DEC-004)。
   * ※入室が成功したコミュニティIDは、ブラウザの **LocalStorage に自動保存**され、次回以降ドロップダウン候補に表示されます。
   * ※コミュニティへの「参加順」に、コミュニティ内で一意となる `1` から始まる連番の**ユーザー番号**が自動で発行されます (DEC-014)。
5. **メイン画面遷移**: 接続完了後、3ペインのメイン画面を表示します。
   * **個別チャット一覧の表示形式**: メンバーは `# [コミュニティ内ユーザー番号] [ニックネーム]`（例: `#12 たろう`）の形式で表示されます。

### 3.2. 発話/送信フロー (送信)
1. **宛先選択**: 左ペイン（複数選択可）または中央ペイン（単一選択）から送信先をチェックします。
   * ※誰も選択されていない場合、新規送信（グループ/個別）をブロックし、UI上に警告を表示します。
2. **チャット履歴表示**: 該当する `ChatHistoryID` の履歴（または新規のクリア状態）を右ペインに表示します。
3. **メッセージ作成の選択**: ユーザーは以下のいずれかの方法でメッセージを作成・送信します。
   * **パターンA：音声発話（録音）**
     * チャットウィンドウの「発話」ボタンをクリックします。
     * 録音ダイアログを表示し、ユーザーの発話を録音します（レベルメーター表示含む (DEC-006)）。
     * ダイアログには「15秒からのカウントダウンタイマー」が表示され、15秒経過すると自動的に録音を停止します。
     * バックエンド登録時、Chrome標準の Web Speech API (SpeechRecognition) でディクテーションされた文字がプレビュー表示されます。
     * 「録音を停止する」ボタンで一時停止し内容を確認でき、その後「送信」ボタンで音声ファイルとテキストがバックエンドに送信されます。録音中に直接「送信」を押すことも可能です。
   * **パターンB：テキスト入力送信** (DEC-018)
     * チャットウィンドウ下部のテキスト入力エリアに、キーボードからテキストを直接入力します。
     * 「送信」ボタン（またはEnterキー）を押下して送信します。
     * 音声データ（`audio_url`）は空で、テキストデータのみがバックエンドに登録されます。
   * **パターンC：発言の取り消し**
     * 自分が送信したメッセージバブルの右端にある「🗑️（ゴミ箱アイコン）」をクリックします。
     * 「発言を取り消しますか。」という確認ダイアログが表示され、「OK」を押すと取り消しが実行されます。
     * 取り消された発言は、音声ファイルがサーバーから削除され、テキスト内容がクリアされます。
     * 画面上ではチャットバブルの痕跡だけが残り、「この発言は取り消しされました。」というグレーアウトされたテキストが表示されます。


### 3.3. 受信フロー (再生)
1. **待機状態**: 受信者はアプリを開いた状態で待機します。
2. **着信・自動再生（グローバル再生）**:
   * 個人設定の「着信メッセージの即時自動再生」がON（デフォルトはON）の場合、**現在表示していない、または現在接続していない他のコミュニティのチャットからの着信であっても、自動的にスピーカーから音声が再生されます** (DEC-009, DEC-014)。
   * **再生動作の分岐**:
     * **音声メッセージの場合**: 音声ファイルを読み込み、そのままスピーカーから再生します。
     * **テキストメッセージの場合**: ブラウザ標準の **Web Speech API (SpeechSynthesis)** を使用して、入力テキストを音声合成で自動読み上げします (DEC-018)。
   * 複数のグループからほぼ同時にメッセージが届いた場合は、届いた順に「再生キュー（Queue）」に格納され、順次再生されます (DEC-003)。
   * ※自身が「録音中」の場合は、再生キューの処理を一時停止（ポーズ）し、録音終了後に再生を再開することで録音への音声混入を防ぎます。
3. **再生中インジケータ**:
   * 音声またはTTSが流れている間、左ペイン（ユーザー）、中央ペイン（グループ）の該当項目の横に「🔊 再生中」インジケータを表示し、今誰の声（または読み上げ）が流れているかを視覚的に明示します。
4. **未読・既読の管理 (耳と目の分離)** (DEC-010):
   * 音声が自動再生されても、ユーザーがそのチャット画面を開いて表示するまでは、チャット履歴一覧の該当ユーザー/グループは**「未読あり（未読バッジ等）」**状態を維持します。
   * ユーザーが未読 of 相手またはグループをクリックし、**チャット履歴を右ペインに表示した時点で「既読」**状態になります。
5. **履歴確認 & 聞き逃し時の再再生**:
   * チャットウィンドウには、文字起こしされたテキスト（または入力されたテキスト）がメッセージバブルとして表示されます。
   * メッセージバブル内には「再生 (▶)」ボタンが備わっており、クリックすることで何度でも音声を再再生できます（テキストメッセージの場合は、再度 **SpeechSynthesis** による読み上げを実行します）。

### 3.4. 招待リンクからの参加フロー (DEC-015, DEC-016)
1. **リンクアクセス**: 招待されたユーザーが `https://[domain]/?c=[community_slug]&code=[invite_code]` にアクセスします。
2. **認証チェック**:
   * **未ログイン状態**: Googleログインを求める画面を表示し、ログイン成功後に手順3へ進みます。
   * **ログイン済状態**: 直接手順3へ進みます。
3. **参加同意ダイアログの表示**:
   * 画面上に**「コミュニティ [コミュニティ表示名] に参加しますか？」という参加意思を問う同意ダイアログ**を表示します。
   * ※初回ユーザーの場合は、ここでニックネームの入力も同時に求めます。
4. **接続・参加完了**: ユーザーが「参加する」を明示的にクリックした時点で、初めてそのコミュニティのメンバーとして登録（ユーザー番号発行）され、入室成功として **LocalStorage にコミュニティIDが保存**され、自動再生のロックが解除された状態でメイン画面へ遷移します。

### 3.5. コミュニティ退出フロー (DEC-016)
1. **退出アクション**: ユーザーがヘッダーのコミュニティ設定メニューから「コミュニティを退出する」をクリックします。
2. **確認ダイアログ**: 「本当に退出しますか？過去の会話履歴が閲覧できなくなります」という警告を伴う確認ダイアログを表示します。
3. **削除処理 (トランシーバー仕様の完全クリーンナップ)**:
   * **管理者権限のクリーンナップ処理**（`leave_community_and_cleanup`関数）が走り、以下のデータが一括削除されます。
   * 該当コミュニティにおいて、自分が含まれていた**すべてのチャットルーム（個人・グループ問わず）が完全に消去**されます。
   * その結果、ルーム内に存在していた**自分および他者のすべてのメッセージ（テキストおよび音声ファイル）も道連れで完全削除**されます。`chat_rooms` 削除に伴い `messages` が FK CASCADE で削除され、その各行の DELETE が Edge Function (`delete-audio-on-message-delete`) を発火させ、Storage 上の音声ファイルも自動削除されます。
   * ※本アプリは「ディープな話に入る前のトランシーバー」という思想のため、退出時に履歴を未来へ残さず完全消去します。
   * データベース（Supabase）の `community_members` テーブルから該当レコードを削除します。
   * **ブラウザの LocalStorage の入室履歴から、該当コミュニティIDを削除します。**
   * ※プッシュ通知（FCMトークン）は端末×ユーザー単位（コミュニティに非依存）のため、**退会では解除しません**（他コミュニティの通知を止めないため）。
4. **メイン画面のリセット**: 画面が未接続（空白）状態へ戻り、コンボボックスの入力履歴リストからも該当コミュニティが削除されます。

### 3.6. ログアウトフロー
1. **ログアウトアクション**: 環境設定の「アカウントからログアウトする」をクリックします。
2. **プッシュ通知の解除**: FCM登録済みの場合、ローディング表示中に **FCMトークンを解除**します（`register-fcm-token` の `unregister`）。セッションが必要なため `signOut()` の前に `await` で実行し、失敗してもログアウトは続行します。
3. **非破壊の切断**: コミュニティからは**切断のみ**を行い、**チャット・グループ・メッセージは一切削除しません**（退会とは異なる）。LocalStorage の入室履歴も保持します。
4. **サインアウト**: Supabase Auth からサインアウトし、未ログイン状態の初期画面へ戻ります。

## 4. 非機能要件・技術要件
* **認証**: Supabase Auth (Google Auth (DEC-005) ＋ Magic Link / メールリンク (DEC-028))
* **リアルタイム通信**: 音声データおよび文字起こしデータのリアルタイム送受信 (Supabase Realtime)
* **オンラインステータス管理**: Supabase Realtime Presence を使用し、異常終了時も正確に在席状態を監視・同期します。
* **音声機能**: 
  * **音声認識 (ディクテーション)**: Web Speech API (SpeechRecognition)
  * **音声合成 (TTS / テキスト読み上げ)**: Web Speech API (SpeechSynthesis) (DEC-018)
  * **録音最適化**: 音声データ通信量削減のため 32kbps を指定し、出だし無音量フェードインを防ぐためAGC（Auto Gain Control）を無効化しています。

## 5. 新着メッセージ通知機能
バックグラウンドや他アプリ利用中に新着メッセージを受信した際、以下の手段で通知を行います（Edge Function経由で送信されます）。
* **Webプッシュ通知 (FCM)**
  * Android、PCなどPWA対応のモダンブラウザで利用可能な標準プッシュ通知です。
  * 「環境設定」から通知を許可することでデバイストークンが登録されます。
* **Discord Webhook 通知 (代替手段)**
  * Appleの仕様上、iOS（iPhone/iPad）等のブラウザではPWA化（ホーム画面追加）しない限りWebプッシュ通知が動作しないという強い制限があります。
  * その代替策として、ユーザーが「環境設定」から個別にDiscord Webhook URLを登録できるようにしています (DEC-022)。
  * 設定した場合、FCMプッシュ通知と並行してDiscordの指定チャンネルにも新着メッセージとアプリへのダイレクトリンクが送信されます。

## 6. ハンズフリー / 操作補助機能

### 6.1. TTT (Trigger word To Talk) ハンズフリーモード (DEC-029)
* 環境設定でウェイクワード（合言葉）を**正規表現**として登録します（未設定時はTTTを有効化できません）。
* チャット画面下部のTTTスライダートグルをONにすると、SpeechRecognition でウェイクワードを常時待ち受けます。待機中はウェイクワード認識オーバーレイを表示します。
* ウェイクワードを検知すると、フラッシュ演出とともに録音を自動開始します。**15秒タイムアウト時のみ自動送信**し、手動停止した場合は送信しません。
* バックグラウンド復帰時（`visibilitychange`）に待機を自動再開します。Android Chrome 固有の SpeechRecognition 挙動にも対応します。

### 6.2. 常時表示 (Screen Wake Lock) (DEC-026)
* ヘッダーの☀️ボタンで画面の自動消灯・ロックを抑止します（デフォルトOFF、リロードで解除）。
* `navigator.wakeLock` 対応環境でのみボタンを表示します。タブ非表示で解放されたセンチネルは画面復帰時に自動再取得します。

### 6.3. メディアボタンPTT (DEC-027)
* ヘッダーの🎧ボタンをONにすると、イヤフォン等の再生/停止（Media Session）ボタンで PTT 操作（1回目=録音開始、2回目=停止＋送信）ができます。
* 対応環境のみ表示、フォアグラウンド前提の機能です。メディアキー配送のため、ON中は微小音量ノイズの連続ループ再生が必須です（完全無音化は不可）。

## 7. 外観・環境設定

### 7.1. 録音モード
* 「両方（音声＋文字おこし）」「録音のみ」「文字おこしのみ」から選択できます。※モバイルでは録音と文字おこしの併用は不可。

### 7.2. 疑似コールサイン音
* 着信メッセージ再生の前後に効果音を鳴らすトランシーバー風演出です（デフォルトON、環境設定でON/OFF）。

### 7.3. カラーテーマ
* ダークモード（標準）とライトモードを切り替えられます。

### 7.4. 背景画像（ぼかし対応）
* 任意の画像ファイルをチャット背景に設定できます。画像はサーバーではなく**ブラウザの IndexedDB にローカル保存**されます（`ChatransceiverDB` / `backgroundStore`）。
* 表示方法（`cover`：全体を埋める／`contain`：全体を表示）と、背景ぼかしのON/OFF（`bg-blur`）を選択できます。表示方法・ぼかし設定は localStorage に保存されます。
