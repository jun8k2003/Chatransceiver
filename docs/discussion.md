### 2. コミュニティの「退出処理」におけるデータベース整合性の確保
* **ステータス**: 議論中 (方針決定: DEC-016)
* **検討ポイント**:
  * **カスケード（連鎖）処理**:
    * `community_members` からレコードが削除された際、そのユーザーがコミュニティ内で所属していたチャットルーム（`chat_room_members`）からも自動的に除外されるようなPostgreSQLのトリガー（Trigger）または外部キー制約（ON DELETE CASCADE）の設定。

### 3. リアルタイム監視（接続数）の設計：複数チャット監視 vs 自分専用Inbox監視
* **ステータス**: 議論中
* **検討ポイント**:
  * アプリ動作時、新着メッセージを検知するためのSupabase Realtimeの監視アプローチ。
  * **アプローチA：複数チャットルーム（ChatHistoryID）の並列監視**
    * クライアントが所属するすべての部屋（`room_id`）の更新を並列で監視する。
    * *課題*: 所属グループ数が増えるにつれて監視接続（Channel）が増え、Supabaseの無料枠制限やクライアントの負荷が増加する。
  * **アプローチB：自分専用の「Inbox（受信箱）」テーブルの監視（Fan-Out-on-Write）**
    * 各ユーザーに紐づく `user_inboxes` テーブルを用意し、クライアントは「自分のIDのInboxレコード追加」のみを**1つの接続で常時監視**する。
    * *仕組み*: メッセージが送信された際、送信側（またはDBのトリガー）が、宛先ルームに所属する全員の `user_inboxes` に「新着メッセージID」を書き込む。
    * *メリット*: 監視接続が常に1つで済むため、負荷が劇的に低く、無料枠での運用に最適。未読・既読管理も「Inboxレコードの有無や既読フラグ」でスマートに実装できる。

### 4. 音声合成 (SpeechSynthesis) の読み上げ制御と連続再生キュー

* **ステータス**: 議論中
* **検討ポイント**:
  * **ブラウザ標準TTSの挙動**:
    * `window.speechSynthesis` によるテキスト読み上げは、デバイスやブラウザによってデフォルトの声（音声エンジン）が異なります。日本語の読み上げを最も安定して行うための `SpeechSynthesisUtterance` のボイス選定（言語設定: `ja-JP` などの指定）の設定。
  * **グローバル再生キューへの統合**:
    * 音声ファイル（Audio要素の `ended` イベント検知）と、テキスト読み上げ（`SpeechSynthesisUtterance.onend` イベント検知）の両方をシームレスに混在させてグローバルキューで連続再生するための統一インターフェースの設計。
