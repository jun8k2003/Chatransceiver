# 待機状態（Standby Mode）機能 計画書

作成日: 2026-06-11

---

## 概要

スマートフォン（Android）でアプリがバックグラウンドになったり画面ロックされた状態でも、
イヤフォンのボタン操作で音声の録音・送信ができる「待機状態」機能を実装する。

---

## 意思決定ログ参照

本機能は DEC-025 として decisions.md に記録する（実装決定後）。

---

## 対象環境

| プラットフォーム | 対応 | 理由 |
|----------------|------|------|
| Android Chrome (PWA / ブラウザ) | ✅ 対応 | Media Session API・バックグラウンド音声再生が安定 |
| iOS Safari | ❌ 対象外（現時点） | バックグラウンドからのマイク操作が技術的に不安定 |
| デスクトップ (PC) | ⚠️ 対象外 | 画面ロックの概念が異なるため待機状態の恩恵が薄い |

---

## 機能要件

### 1. 待機状態のON/OFF

- 待機状態はデフォルトでOFF
- ユーザーが能動的にONにする操作が必要（UIにトグルを設ける）
- ONにできる条件：Androidデバイスと判定された場合のみ
- iOSまたはPC環境でトグルを操作しようとした場合、「この環境では利用できません」と表示してONにできないようにする
- 待機状態中はUI上に目立つインジケーター（例: オレンジのバッジ or 常時表示テキスト）を表示する

### 2. バックグラウンドでのインスタンス維持

- 待機状態ON時、OSにメディア再生中と認識させるため以下を行う:
  - **Media Session API** でメタデータ（アプリ名、アイコン）を設定し再生状態を「playing」に維持する
  - **AudioContext** で超低音量（gain ≈ 0.001）のホワイトノイズを **連続ループ再生**する
    - `AudioBufferSourceNode` + `loop: true` で途切れなく再生
    - 2秒ごとの断続再生ではなく連続再生とすること（断続は隙間をOSに停止と判断される）
  - バックグラウンド時も `AudioContext.state` が `running` であることを監視し、`suspended` になったら `resume()` を試みる

### 3. マイクの事前取得

- 待機状態をONにする操作時（フォアグラウンド・ユーザージェスチャー確定のタイミング）に
  `getUserMedia({ audio: true })` を呼び出してマイクストリームを事前取得・保持する
- 録音開始時は保持済みストリームを使用し、バックグラウンドからの `getUserMedia()` 新規呼び出しを行わない
- 待機状態をOFFにした場合は取得済みストリームを解放する（`track.stop()`）

### 4. イヤフォンボタンによる操作

- Media Session API の `setActionHandler` を用いてイヤフォンの再生/停止ボタン（センターボタン）を捕捉する
- **1回目押下 (play または pause アクション)**: 録音開始（PTT録音と同じ処理）
- **2回目押下**: 録音停止 → 送信（既存の PTT 送信フローへ渡す）
- 録音中であることを示すため、アクション名を `"録音中..."` に変更する（Media Session の `title` 更新）
- 録音中に別のアクション（たとえばOSによる割り込み）が発生した場合は録音をキャンセルする

### 5. 操作フィードバック用ビープ音

画面が見えない状態での操作確認のため、録音の開始・終了それぞれに異なるビープ音を再生する。

- **録音開始ビープ**: 低音 → 高音の2段階トーン（例: 800Hz 80ms → 1200Hz 80ms）
  - 「準備完了、今しゃべれる」を直感的に伝える上昇音
- **録音停止・送信ビープ**: 高音 → 低音の2段階トーン（例: 1200Hz 80ms → 800Hz 80ms）
  - 「送信した、終わった」を伝える下降音
- ビープ音は **Web Audio API (`OscillatorNode`)** を用いてアプリ内で合成する（外部音声ファイル不要）
- ビープ音の音量はホワイトノイズループとは独立した gain ノードで制御し、ループ音量に影響されない
- ビープ音の再生はバックグラウンド状態でも同じ AudioContext 上で動作するため、追加の対策は不要
- 将来的にビープ音のON/OFFや音量を設定できる余地を設計上残しておく（初期実装では常時ON）

### 6. Android判定ロジック

```
UserAgent に "Android" が含まれる
  AND
navigator.mediaSession が利用可能
  AND
AudioContext が利用可能
```

上記すべてを満たす場合にAndroid対応環境と判定する。
判定結果はUI初期化時に1度だけ評価する。

---

## 非機能要件

- 待機状態中でもフォアグラウンドでの通常のPTT操作は引き続き動作する
- 連続ループ音声はユーザーに聞こえないこと（gain ≦ 0.001）
- 待機状態中にバッテリー消費が増加することをUI上で警告する（ONにする際に1度表示）
- 待機状態は画面リロードで解除される（意図的：リロードはユーザーの能動的操作）

## StandbyService の主なインターフェース（案）

```typescript
class StandbyService {
  // 待機状態をONにする（マイク取得・AudioContext起動・Media Session設定）
  async activate(): Promise<void>

  // 待機状態をOFFにする（ストリーム解放・AudioContext停止）
  deactivate(): void

  // 現在の状態
  get isActive(): boolean

  // イヤフォンボタン押下時のコールバック設定
  onButtonPress(callback: () => void): void

  // Android対応環境か判定（static）
  static isSupported(): boolean
}
```

## 未解決事項・リスク

| 項目 | 内容 |
|------|------|
| Androidでのマイク維持 | バックグラウンドでOSがマイクストリームを強制停止するケースがある。対策として activate 時にストリームの `ended` イベントを監視し、ユーザーに再取得を促すトーストを表示する |
| 2ステート管理 | イヤフォンボタンの1回目/2回目の「録音中かどうか」状態がOSの再生/停止状態とずれる可能性。Media Session の `playbackState` を状態の正とするか、アプリ内フラグを正とするかを実装時に決める |
| 他アプリによる音声割り込み | 電話着信などで AudioContext が `suspended` になった場合、録音中であれば自動キャンセルしてユーザーに通知する |
| Chrome以外のAndroidブラウザ | Firefox for Android などは Media Session の挙動が異なる場合がある。初期対応は Chrome のみとし、他ブラウザは「非対応」として扱う |

## 実装時の進め方（提案）

1. `StandbyService` の単体実装と Android 判定テスト
2. 設定モーダルへのトグル追加（Android 以外は disabled 表示）
3. 待機状態中のインジケーター表示
4. イヤフォンボタン → PTT 処理の接続
5. バックグラウンド状態でのマイクストリーム監視・復帰ロジック
6. 実機（Android Chrome）での動作確認
